**💬 面试题目与回答要点**

**1. MVC 与 MVVM 的区别**

**Q：你用过 MVC 架构，也用过 MVVM 架构，能说一下这两个有什么区别吗？**

> **A：**MVC 在 Android 中耦合度较高：Activity 同时承担 UI
> 绘制、生命周期管理和业务逻辑，职责不清晰。MVVM 通过 ViewModel 将 Data
> 层与 UI 层解耦，使用数据驱动更新 UI，更适合 Jetpack Compose
> 响应式开发，代码更清晰，也便于测试。

**2. Kotlin 协程与线程的区别**

**Q：在 Kotlin 里，协程和线程有什么区别？**

> **A：**线程是系统内核级别的，是 CPU
> 调度的基本单位，由操作系统负责切换，开销较大；协程是用户态的轻量级线程，由开发者/运行时自行调度，创建和切换成本低。线程间通信主要依赖共享内存+锁，协程间通信则通过
> Channel 或开发者定义的共享状态变量实现，更安全简洁。

**3. Compose 性能优化 / 重组问题**

**Q：你遇到的性能最卡顿的地方是什么，你是怎么解决的？**

> **A：**主要问题是 Compose
> 不必要的重组（Recomposition）。原因一：自定义 Model
> 类不稳定，导致频繁重组------解决方案是使用 \@Stable / \@Immutable
> 注解声明其稳定性，或通过 Compose
> 编译器插件检测。原因二：组合函数粒度过大、状态范围过宽------使用状态提升（State
> Hoisting）将状态上移到
> ViewModel，将事件以函数参数形式层层上传，遵循单向数据流原则，缩小重组范围，提升性能。

**4. Clean Architecture 三层架构**

**Q：简历里写的 Clean Architecture
三层分别是什么？这样分层有什么好处？**

> **A：**三层分别是：Data 层（数据层）、Domain 层（领域/业务逻辑层）、UI
> 层（界面展示层）。好处：Data 层专注数据来源（本地 Room + 远端
> API），采用离线优先策略；Domain 层封装业务逻辑（如创建帖子的
> UseCase），对外屏蔽数据细节；UI 层只负责展示数据、响应 ViewModel
> 状态。各层职责单一，便于独立修改和测试，逻辑清晰，耦合度低。

**5. OkHttp 与 Retrofit 的区别**

**Q：OkHttp 和 Retrofit 有什么区别？**

> **A：**Retrofit 是对 OkHttp 的封装，提供更高层的抽象：支持添加
> Converter Factory（如
> GsonConverterFactory），直接将响应解析为数据模型，无需手动处理
> JSON；接口声明式写法更优雅，可扩展性强。OkHttp 更底层，主要负责实际的
> HTTP 请求执行，可通过 Interceptor 拦截请求/响应，实现统一添加
> Token、统一错误处理等逻辑。Retrofit 内部需要一个 OkHttpClient
> 实例来执行请求。

**6. 自定义 Gradle 插件的好处**

**Q：自定义 Gradle 插件有什么好处？**

> **A：**好处有三点：①逻辑复用------将各模块通用的构建配置提取到插件中，新建模块时直接复用，避免重复代码；②依赖统一------所有模块使用同一套插件，依赖版本统一管理，不会出现版本冲突问题；③简化模块化开发------多模块项目中，每个模块的构建脚本更简洁，维护更方便，省时省力。

**7. Flutter 与 Kotlin/Compose 的对比体验**

**Q：你做过 Flutter 开发，对比 Kotlin/Compose，哪个更好用？**

> **A：**个人体验来看，Flutter 的状态管理方案繁多（Provider / Riverpod /
> Bloc 等），对初学者而言选择成本高，上手比 Compose 费力。Compose
> 的设计更聚焦：只追踪 State 变化驱动
> UI，重组范围可控，底层设计理念更清晰；而 Flutter
> 的重绘机制相对不够精细可控。不过 Flutter 跨平台能力更强，UI
> 一致性更好；各有适用场景。

**8. 冒泡排序的时间/空间复杂度**

**Q：冒泡排序的算法复杂度是多少？**

> **A：**时间复杂度：O(n²)，每次将最大元素交换到末尾，需要 n-1
> 轮遍历。空间复杂度：O(1)，原地排序，不需要额外的数组空间，只用到常数级辅助变量。

**9. Kotlin 内存管理（JVM GC）**

**Q：Kotlin 是怎么做内存管理的？**

> **A：**Kotlin 运行在 JVM（或 Android 的 ART/Dalvik）之上，内存管理由
> GC（垃圾回收器）负责。Android 上的 ART
> 使用分代回收策略，常见算法包括标记-清除、复制算法等。具体回收细节偏底层，日常开发关注点更多在于避免内存泄漏（如持有
> Context 引用、非静态内部类、注册未注销等）。

**10. 哈希表的原理**

**Q：哈希表的原理是什么？**

> **A：**哈希表通过将 key 经过哈希函数转换为
> hashCode，映射到数组的对应桶（bucket）位置实现快速存取。由于不同 key
> 可能映射到同一位置（哈希冲突），需要冲突解决策略：Java 7
> 采用数组+链表；Java 8 优化为数组+链表+红黑树，当链表长度超过 8
> 时转为红黑树，使查找时间复杂度从 O(n) 降为 O(log n)。
