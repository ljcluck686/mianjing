### 1. 进程、线程、协程是什么？协程是怎么实现的？

#### （1）定义与核心区别

- **进程**：操作系统资源分配的基本单位（如内存、CPU 时间片），每个进程有独立的地址空间，进程间切换开销大（需内核参与）。

- **线程**：操作系统调度的基本单位，依赖进程存在，同一进程内的线程共享地址空间，切换开销比进程小，但仍需内核上下文切换。

- **协程**：用户态的轻量级 “线程”，完全由程序控制调度（无需内核参与），切换开销极小，可在单个线程内实现多任务 “并发”。

#### （2）协程的实现原理

协程的核心是 “用户态上下文切换”，通过保存 / 恢复程序执行状态（如寄存器值、程序计数器、栈指针）实现任务切换，关键技术点如下：

1. **上下文存储**：用结构体存储协程的执行状态（如ucontext_t在 C 语言中的应用，或 Kotlin 协程中的Continuation）；

1. **切换触发**：主动调用切换接口（如yield），将当前协程状态保存到结构体，再加载目标协程的状态；

1. **调度模型**：分 “有栈协程”（独立栈空间，切换时保存完整栈）和 “无栈协程”（依赖编译器优化，用状态机实现，如 Kotlin 协程）；

1. **依赖支持**：需语言或框架层面实现调度器（如 Python 的gevent、Kotlin 的CoroutineDispatcher），管理协程的创建、切换与销毁。

#### （3）简单示例（Kotlin 协程）

```
// 通过launch创建协程，Dispatchers.Main指定主线程调度
lifecycleScope.launch(Dispatchers.Main) {
    // 耗时操作切到子线程（withContext主动切换协程上下文）
    val data = withContext(Dispatchers.IO) { 
        api.fetchData() // 耗时网络请求
    }
    // 切回主线程更新UI（协程自动恢复上下文）
    tvContent.text = data
}
```

### 2. Binder 跨进程通信（IPC）原理

#### （1）Binder 的核心角色

- **客户端（Client）**：发起 IPC 请求，持有服务端的 “Binder 代理对象”（Proxy），通过代理调用服务端方法；

- **服务端（Server）**：提供服务，继承Binder类，重写onTransact方法处理客户端请求；

- **ServiceManager**：“服务注册表”，管理服务端的 Binder 对象，客户端需先向其查询服务端代理；

- **Binder 驱动**：内核层组件，负责转发请求、管理进程间数据拷贝（仅一次拷贝，比 Socket 高效）。

#### （2）通信流程（以 “客户端调用服务端获取数据” 为例）

1. **服务端注册**：服务端创建 Binder 对象，通过addService向 ServiceManager 注册（告知服务名称与 Binder 地址）；

1. **客户端获取代理**：客户端通过getService向 ServiceManager 查询服务，ServiceManager 返回 Binder 代理对象（Proxy）给客户端；

1. **客户端发起请求**：客户端调用 Proxy 的方法，Proxy 将参数序列化（Parcel）后，通过transact方法通知 Binder 驱动；

1. **驱动转发与唤醒**：Binder 驱动将请求转发给服务端的 Binder 对象，若服务端未唤醒则唤醒服务端线程；

1. **服务端处理请求**：服务端线程执行onTransact方法，反序列化参数、处理逻辑，将结果序列化后通过驱动返回；

1. **客户端接收结果**：Binder 驱动将结果转发给 Proxy，Proxy 反序列化后返回给客户端。

### 3. Handler 中跨线程通信是怎么实现的？

#### （1）核心组件与关系

- **Handler**：发送（sendMessage/post）和处理（handleMessage）消息的 “工具类”，需绑定 Looper；

- **Looper**：“消息循环器”，循环从 MessageQueue 取消息，分发给对应 Handler（每个线程最多 1 个 Looper）；

- **MessageQueue**：“消息队列”，存储 Handler 发送的 Message，按时间戳排序（阻塞队列，无消息时线程休眠）；

- **Message**：通信载体，存储消息内容、目标 Handler、执行时间等。

#### （2）跨线程通信流程（以 “子线程向主线程发消息更新 UI” 为例）

1. **主线程初始化 Looper**：Android 主线程（UI 线程）在启动时自动执行Looper.prepareMainLooper()，创建 Looper 和 MessageQueue，再调用Looper.loop()开启消息循环；

1. **主线程创建 Handler**：主线程中创建 Handler，重写handleMessage（用于接收子线程消息并更新 UI），此时 Handler 会绑定主线程的 Looper；

1. **子线程发送消息**：子线程通过主线程 Handler 的sendMessage发送 Message，Message 会被加入主线程的 MessageQueue；

1. **Looper 分发消息**：主线程 Looper 循环调用MessageQueue.next()取消息，若有消息则调用message.target.dispatchMessage()（target即绑定的 Handler）；

1. **Handler 处理消息**：Handler 的dispatchMessage调用handleMessage，在主线程中执行 UI 更新逻辑，完成跨线程通信。

#### （3）关键注意点

- 子线程创建 Handler 需手动调用Looper.prepare()（创建 Looper）和Looper.loop()（开启循环），否则会报 “Can't create handler inside thread that has not called Looper.prepare ()”；

- MessageQueue 的阻塞依赖epoll机制（Linux），无消息时线程休眠，有消息时被唤醒，避免 CPU 空转。

### 4. 播放器底层原理

播放器底层核心是 “音视频数据的获取→处理→渲染” 流程，关键步骤如下：

1. **数据获取（解协议）**：从本地文件或网络（HTTP/RTMP/HLS）获取音视频数据流，处理网络超时、断连等问题（如用 FFmpeg 的avformat_open_input打开流）；

1. **解封装（Demuxing）**：将封装格式（如 MP4/FLV/MKV）的数据流分离为独立的音频流（如 AAC）和视频流（如 H.264），获取音视频的编码信息、时间戳（PTS/DTS）（FFmpeg 的avformat_find_stream_info+av_read_frame）；

1. **解码（Decoding）**：将压缩的音视频数据（如 H.264→YUV，AAC→PCM）解码为原始数据，需兼容不同编码格式，硬件解码（如 Android 的 MediaCodec）比软件解码（FFmpeg）效率更高；

1. **音视频同步**：根据 PTS（显示时间戳）调整音视频播放速度，避免 “音画不同步”，常用策略：以音频 PTS 为基准，调整视频播放速度（如丢帧或重复帧）；

1. **渲染（Rendering）**：

- - 视频：将 YUV 原始数据转换为 RGB，通过 OpenGL ES 或 SurfaceView 渲染到屏幕；

- - 音频：将 PCM 原始数据通过 AudioTrack（Android）或 ALSA（Linux）输出到扬声器。

#### （4）核心框架示例

常用底层框架组合：FFmpeg（解协议、解封装、软件解码）+ MediaCodec（Android 硬件解码）+ OpenGL ES（视频渲染）+ AudioTrack（音频播放）。
